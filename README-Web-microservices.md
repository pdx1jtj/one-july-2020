<h2><a id="user-content-introduction" class="anchor" aria-hidden="true" href="#introduction"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Introduction</h2>
<p>In the web-tier microservices workload, a simulated user logs in to a web application that does three things: provides a selection of stock options, performs Monte-Carlo simulations with those stocks, and presents the user with options that may be of interest. The workload reports performance in transactions per second, which testers can use to directly compare IaaS stacks and to evaluate whether any given stack is capable of meeting service-level agreement (SLA) thresholds.</p>

<h3><a id="user-content-configure-the-benchmark" class="anchor" aria-hidden="true" href="#configure-the-benchmark"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Configure the benchmark</h3>
<p>The installation scripts within the installation directory will install and create a Kubernetes cluster using Kubespray. They will help you:</p>
<ul>
<li>configure your environment to run CloudXPRT,</li>
<li>get the IP addresses for all machines in your cluster,</li>
<li>configure passwordless SSH,</li>
<li>install Ansible/Kubespray requirements,</li>
<li>create the cluster, and</li>
<li>remove the cluster once you are done running CloudXPRT.</li>
</ul>

<h4><a id="user-content-installation-steps" class="anchor" aria-hidden="true" href="#installation-steps"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Installation steps</h4>
<h5><a id="user-content-configure-the-environment" class="anchor" aria-hidden="true" href="#configure-the-environment"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Configure the environment</h5>
<ol>
<li>
<p>In each machine in your cluster:</p>
<ul>
<li>Set the root password (Note: <strong>must be the same on each machine</strong>)</li>
</ul>
</li>
</ol>
<pre><code>	sudo passwd root
</code></pre>
<pre><code>- Log in as root
</code></pre>
<pre><code>	su
</code></pre>
<pre><code>- Ensure openssh-server is installed
</code></pre>
<pre><code>	apt-get install openssh-server -y
</code></pre>
<pre><code>- Allow Root login access
 Edit /etc/ssh/sshd_config modify the *PermitRootLogin* line to be
</code></pre>
<pre><code>	PermitRootLogin yes
</code></pre>
<pre><code>- Restart sshd
</code></pre>
<pre><code>	service sshd restart
</code></pre>
<h5><a id="user-content-master-node" class="anchor" aria-hidden="true" href="#master-node"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Master node</h5>
<p>Go to the installation directory.</p>
<pre><code>	cd CloudXPRT_vX.XX_web-microservices/installation
</code></pre>
<ol>
<li>
<p>Edit cluster_config.json</p>
<p>For each machine in the cluster, add its IPV4 address and optionally desired hostname. One machine per {..} section within the "nodes" list, starting with the master node.</p>
<p><strong>Note:</strong> Although optional, each hostname must be unique and may only contain <strong>lowercase alphanumeric characters</strong>. If hostnames are not provided, Kubespray will rename each host as node1, node2, ..., nodeN. This means that the master node's hostname will be changed to 'node1'.</p>
<p>If your machines are behind a proxy, make sure to set "set_proxy" to "yes", and configure the settings for "http_proxy" and "https_proxy". Those proxy settings will be applied on all the nodes to ensure that they can communicate through the Kubernetes networking plugin. Furthermore, you must reboot the nodes in order for them to take effect since /etc/environment is modified. You have the option "reboot" to allow the prepare-cluster.sh script to reboot all the nodes automatically. By default, the reboot option in cluster_config.json is set to 'yes'. If you set it to 'no', please manually reboot your machines after running prepare-cluster.sh, otherwise the cluster creation in the step 3 will fail.</p>
<p>Example configuration for a three node cluster:</p>
</li>
</ol>
<pre><code>  "nodes": [
        {
            "ip_address": "192.168.0.11",
            "hostname": "master"
        },
        {
            "ip_address": "192.168.0.12",
            "hostname": "worker1"
        },
        {
            "ip_address": "192.168.0.13",
            "hostname": "worker2"
        }
    ],
</code></pre>
<ol start="2">
<li>In the master node, run "prepare-cluster.sh" script as root to perform preparation steps.</li>
</ol>
<pre><code>	sudo ./prepare-cluster.sh
</code></pre>
<ol start="3">
<li>In the master node, run the "create-cluster.sh" script as root</li>
</ol>
<pre><code>	sudo ./create-cluster.sh
</code></pre>
<pre><code>**Note**:
- This process may take anywhere from 6 up to 20 minutes.
- If you get an error with respect to docker-ce repository '**RETRYING: ensure docker-ce repository public key is installed ...**', double check that the proxies are configured correctly! You may repeat the "prepare-cluster.sh" script to set this again, or you may manually edit them in each node of your cluster.
- You should also double check that the date and time are the same on all of the nodes.

<p>Edit config.json file to set the parameters for CloudXPRT.</p>
<ul>
<li><strong>runoption</strong>: Run only a specific microservice or all of them individually (not in multitenancy)</li>
<li><strong>iterations</strong>: Number of iterations to run for each microservice</li>
<li><strong>hpamode</strong>: If true, use Kubernetes HPA to scale pods as load increases. Otherwise, create max pods from the beginning</li>
<li><strong>postprocess</strong>: If true, run the postprocess binary at the end of the run</li>
<li><strong>ppoutputfile</strong>: Sef if want post process results to be saved in a file</li>
<li><strong>autoloader.initialclients</strong>: The initial number of clients the load generator will create</li>
<li><strong>autoloader.clientstep</strong>: The number of clients to increase for each load generator iteration</li>
<li><strong>autoloader.lastclient</strong>: Stops the load generator when reaching this number of clients. If set to -1, it will continue to run until cluster is saturated (i.e. CPU ~100%)</li>
<li><strong>autoloader.SLA</strong>: Service Level Agreement for 95 percentile latency constraint. If set to -1, there is no latency constraint</li>
<li><strong>autoloader.timeinterval</strong>: The amount of time to spend within each load generator iteration with the specified amount of clients</li>
<li><strong>workload.version</strong>: Docker image version to use</li>
<li><strong>workload.cpurequests</strong>: Amount of CPU cores requested to assign to each pod (integer only)</li>
</ul>
<p><strong>Note</strong>: cnbrun, gobench, autoloader, and all shell scripts need to have executable permissions</p>

<p>Instructions for running the benchmark in demo mode with UI</p>
<p>These scripts make it easy to bring up all of the services that are used during a normal CloudXPRT run. The main difference is that only one replica of each service is deployed and the services remain deployed until you want to remove them. It gives users time to interact with the web pages that the web server is serving.</p>
<h3><a id="user-content-deploy-all-services" class="anchor" aria-hidden="true" href="#deploy-all-services"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Deploy all services</h3>
<pre><code>	sudo ./services.sh up
</code></pre>
<h3><a id="user-content-viewing-web-server-pages" class="anchor" aria-hidden="true" href="#viewing-web-server-pages"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Viewing Web Server Pages</h3>
<p>Once all of the services are up, the script will print out possible address you can visit to interact with the front end. Example output from the script:</p>
<pre><code>   You may access the web server UI by visiting one of the following addresses in your web browser:
	   http://10.233.47.78:8070 on any machine within the cluster, or
	   http://192.168.0.11:30800 externally on any machine within the same network
</code></pre>
<p>The second address printed is the ip address of the master node. If you have a multi-node cluster, you can access the web service by visiting the ip address of either node along with the same port number listed from the script.</p>
<p>To get the web-service ClusterIP address and ports exposed:</p>
<pre><code>	kubectl get service web-service
		NAME          TYPE       CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
		web-service   NodePort   10.233.47.78   &lt;none&gt;        8070:30800/TCP   14m
</code></pre>
<p>The ClusterIP address and port 8070 is only accessible from any nodes within Kubernetes cluster.</p>
<p>Externally, use the actual node's ip address along with the port listed within the 30000-32767 range.</p>
